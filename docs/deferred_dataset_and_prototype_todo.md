## Prototype architecture follow-ups

- [x] Keep the prototype wired to generated exports so a final definitive dataset can replace the current baseline without redesigning the frontend. *(Held throughout — every dataset refresh (final rosters, stats, coaches, referees, confederations) dropped in via regenerated JSON with zero frontend redesign.)*
- [x] Add the typed data-access layer that reads from `data/processed/app_exports/`. Done via `site/src/lib/data/`.
- [x] Decide whether the app should keep direct `@data` imports, move to a runtime fetch model from copied files in `public/`, or use route-level dynamic `import()` code-splitting. **Decision (2026-07-13): keep the direct `@data` imports.** After the beta test on the live URL there is no functional or performance reason to switch: pages load quickly and filters/selections respond without lag on the devices it was tested on. The single downside is bundle size — the production JS is now one ~4.2 MB (≈596 KB gzip) chunk — but gzip-over-CDN at that size is not a real-world problem for this audience, and the simplicity/type-safety of direct imports is worth keeping. **Switch only if a concrete symptom appears** (e.g. slow first paint on low-end mobile, or the chunk grows materially with images/new datasets). If that happens, the cheapest fix is option 3 below (no fetch/loading-state code, no `public/` copy). This is tracked separately as the "Route-level `import()` code-splitting" enhancement under "Next additions" — it is now an optional enhancement, not an open decision. Options considered:
  1. **Current: direct `@data` imports (chosen).** Simplest code (no loading states, fully typed at compile time), but every route pays for all datasets in one bundle (~4.2 MB / ≈596 KB gzip after Leaflet + Fuse.js).
  2. **Runtime fetch from a `public/`-copied dataset.** Shrinks the JS bundle and lets routes fetch only what they need, at the cost of loading states, fetch error handling, a small build step to copy `data/processed/app_exports/*.json` into `site/public/`, and slightly weaker out-of-the-box type safety.
  3. **Route-level dynamic `import()`.** Keeps the simplicity and type safety of direct imports, but defers each dataset's `import()` to the route that needs it, so Vite/Rollup code-splits it into its own chunk automatically — no fetch/loading-state code and no `public/` copy step required. The best effort-to-benefit tradeoff *if* bundle size ever becomes a real problem.

Birthplace is instead surfaced as text/aggregate charts on `Insights` (birth country/city rankings, home-grown-vs-diaspora), which needs no per-city geocoding.
